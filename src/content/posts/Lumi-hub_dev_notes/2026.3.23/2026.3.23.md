## 摘要

今天的核心成果是把“启动器能力”从外部 C++ 壳逐步内聚到 Flutter 应用本体，形成更接近 QQ 的单入口体验：

1. 打开前端后先进入启动准备页，而不是直接登录页。
2. 启动准备页负责执行环境检测、AstrBot 拉起/复用、Host 连通验证、WebSocket 建链。
3. 启动失败时提供详细可追踪日志，并支持一键打开日志目录。
4. 前端关闭策略可配置：默认不关闭 AstrBot，避免影响用户把 AstrBot 当独立服务使用。

这次改造的关键不是“加一个新页面”，而是将启动编排、生命周期管理、失败可观测性与用户控制权统一到一套流程中。

## 1. 启动流程重构：从“直进登录”到“启动准备页”

### 背景

旧流程里，`WsService` 在初始化时直接尝试连接，UI 层只在登录页被动展示“正在连接服务器...”。这种方式对于“需要先拉起 AstrBot 再连 Host”的场景不友好：

- 用户看不到系统正在执行什么准备动作。
- 失败时缺乏结构化诊断信息。
- 启动责任散落在多个组件，难以维护。

### 方案

引入专门的 `BootstrapService` 负责启动编排，并通过 `BootstrapScreen` 可视化整个阶段状态。

### 关键实现

#### 入口改造（`client/lib/main.dart`）

- Provider 树新增 `BootstrapService`（依赖 `WsService`）。
- `AuthWrapper` 路由逻辑变更：
  - `bootstrap.isReady == false` -> 始终显示 `BootstrapScreen`
  - 准备完成后再走 `AuthScreen / ChatScreen`
- 窗口关闭拦截：`windowManager.setPreventClose(true)` + `WindowListener.onWindowClose()`，用于执行退出策略。

#### 新增启动状态机（`client/lib/services/bootstrap_service.dart`）

定义阶段枚举：

```dart
enum BootstrapStage {
  init,
  checkingEnv,
  checkingHost,
  startingAstrBot,
  waitingHost,
  connectingWs,
  ready,
  failed,
}
```

按阶段串行执行：

1. `checkingEnv`：`python --version` 检测环境可用性。
2. `checkingHost`：探测 `127.0.0.1:8765` 是否已在线。
3. `startingAstrBot`：若未在线，按约定路径执行 `python main.py` 拉起 AstrBot。
4. `waitingHost`：最长 45 秒轮询端口就绪。
5. `connectingWs`：调用 `WsService.connect()` 建立 WebSocket。
6. `ready`：进入登录/自动恢复流程。
7. 任一阶段失败进入 `failed`，并暴露可重试入口。

#### 启动页（`client/lib/screens/bootstrap_screen.dart`）

- 展示阶段提示文案。
- 展示实时日志（滚动）。
- 失败时展示错误容器。
- 提供 `重试启动` 按钮（失败态）。

## 2. 启动失败可观测性增强：日志落盘 + 一键打开目录

### 背景问题

此前启动日志仅存在内存中，用户关闭应用或重启后日志丢失；排障需要复现场景，成本高。

### 方案

在 `BootstrapService` 中实现统一日志落盘：

- 默认日志目录：`%LOCALAPPDATA%\LumiHub\logs`
- 文件：`launcher.log`
- 每次启动 session 写入分隔线：`---------------- new session ----------------`
- 失败时在 UI 明确显示日志文件完整路径

### 实现细节

#### 文件初始化

在 `start()` 的最前面调用 `_ensureLogFileReady()`：

- 首次启动自动创建目录与日志文件。
- 每次启动会追加 session 分隔线。

#### 统一日志管线

`_log()` 同时做三件事：

1. 记录到内存列表（用于 UI 展示）
2. 控制内存日志长度（保留最近 20 条）
3. 异步追加到日志文件（`_appendLogLine`）

这样做到“UI实时可见 + 文件持久化”双通道。

#### 错误信息增强

`_checkPythonAvailable()` 在失败场景会记录：

- 退出码
- stderr 文本
- 异常（无法执行命令）

`_fail()` 会追加：

- 启动失败原因
- 详细日志文件路径

#### 打开日志目录按钮

`BootstrapService.openLogDirectory()` 通过调用系统 `explorer` 打开目录。

`BootstrapScreen` 增加 `打开日志目录` 按钮（非失败态也可用），并在页面底部显示日志文件路径。

## 3. AstrBot 生命周期策略：用户可控的“退出联动”

### 用户场景

用户不一定把 AstrBot 只用于 Lumi-Hub，前端关闭后不应强制关闭 AstrBot。

### 设计原则

默认保守：前端关闭时 **不** 关闭 AstrBot。

仅在以下条件同时满足时才执行关闭：

1. 用户显式开启“关闭前端时同步关闭 AstrBot”
2. AstrBot 是本次由 Hub 拉起（避免误杀用户手工启动的实例）

### 关键实现

#### 设置持久化（`client/lib/services/app_settings.dart`）

- 新增配置项：`app.close_astrbot_on_exit`
- 默认值：`false`
- 使用 `SharedPreferences` 持久化

#### UI 开关（`client/lib/screens/chat_screen.dart`）

在“偏好设置”中新增开关：

- 文案：`关闭前端时同步关闭 AstrBot`
- 说明：`默认关闭。建议保持关闭，避免影响你单独使用 AstrBot。`

#### 关闭钩子（`client/lib/main.dart`）

`onWindowClose()` 中读取设置并调用：

```dart
await bootstrap.handleAppExit(
  closeAstrBotOnExit: settings.closeAstrBotOnExit,
);
```

#### 进程保护逻辑（`client/lib/services/bootstrap_service.dart`）

- 启动 AstrBot 时记录 PID。
- 若开关开启且该 PID 属于 Hub 启动，执行：
  - `taskkill /PID <pid> /T /F`

## 4. 架构知识点与设计考量（Deep Dive）

### 4.1 为什么要用单独的 BootstrapService？

把启动编排放进 `WsService` 会让通信层承担“进程管理 + 环境检查 + UI日志”等跨层职责，导致高耦合。独立服务的收益：

- 职责单一：`WsService` 只管 WebSocket。
- 可测试性更好：启动状态机可独立模拟。
- 失败定位清晰：每个阶段可追踪。

### 4.2 为什么采用阶段状态机而不是布尔变量拼接？

多个布尔值会产生状态组合爆炸（如 `isStarting` + `isFailed` + `isConnected`），难以保证一致性。枚举状态机可做到：

- 单一真状态（Single Source of Truth）
- UI 映射稳定
- 扩展阶段时不破坏旧逻辑

### 4.3 为什么日志写盘用追加而不是覆盖？

启动故障常常需要对比多次尝试，追加日志可以保留历史轨迹；通过 session 分隔线实现低成本分段。

### 4.4 为什么“退出联动”默认关闭？

这是多应用共享后端时的安全默认（Safe Default）：

- 防止误杀用户独立运行的 AstrBot。
- 尊重“Hub 只是 AstrBot 消费者之一”的生态定位。

### 4.5 为什么记录“是否由 Hub 拉起 + PID”？

仅凭端口在线无法判断进程归属，盲目 taskkill 会引发误杀。记录来源与 PID 可以做到“最小影响关闭”。

## 7. 已知边界与后续建议

### 边界

1. 当前日志文件是单文件追加（`launcher.log`），会持续增长。
2. `taskkill /F` 为强制关闭；若后续需优雅停机，建议补一个 Host 侧 shutdown 协议。
3. `ProcessStartMode.detached` 在某些极端环境下可能受权限策略影响。

### 建议

1. 日志滚动策略：按大小/天数归档。
2. 增加“导出本次启动日志”按钮（生成会话专属文件）。
3. 启动页增加“复制诊断信息”按钮，便于用户反馈。
4. 在设置页展示“当前 AstrBot 来源：Hub 拉起 / 外部运行”。
